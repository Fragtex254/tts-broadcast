# TTS Broadcast Content Workbench Design System

## 1. 产品定位

TTS Broadcast 是面向长期使用的 AI 音频与内容生产工作台，不是营销落地页。界面首先帮助用户扫描信息、阅读长文本、判断生成任务状态、试听与比较音色、校对内容，并在多步骤流程中始终理解当前位置、当前任务和下一步动作。

设计语言为 **Warm Workbench / Soft Editorial**：暖纸色提供耐看的工作背景，清晰的工作台结构承载高密度操作，柔和色彩仅表达动作、状态和内容类型。

## 2. 产品设计原则

1. **内容优先**：长文本、音频资产和任务状态高于装饰；减少嵌套卡片和重复标题。
2. **任务可见**：页面首屏说明当前位置、对象状态和首要操作；长任务同时显示阶段、进度、结果或恢复路径。
3. **层级克制**：通过留白、排版、分隔线和背景实度建立层级，不用更多彩色块或阴影解决结构问题。
4. **高频稳定**：导航、筛选、列表选择和播放器不做路由入场或持续动画；反馈即时、短促、可打断。
5. **一致但不僵硬**：同类控件共享尺寸、圆角、边框和状态；阅读区可以比工具区更疏朗。
6. **渐进呈现**：默认显示完成任务所需的信息，次级元数据和危险操作按需展开，但主要内容和主要操作不能依赖 hover 才可理解。

## 3. 色彩语义

组件只使用现有语义 token，不直接散用硬编码颜色。

| Token | 语义 |
| --- | --- |
| `paper` | 应用与阅读背景 |
| `paper-2` | Sidebar、固定工具区、次级底色 |
| `ink` | 标题、主要正文、关键数据 |
| `ink-soft` | 描述、元数据、次级操作；正文至少使用 70% 不透明度 |
| `lemon` | 主操作、等待、需注意状态 |
| `sage` | 完成、保存成功、确认 |
| `lilac` | 编辑、转换、处理中、选中态 |
| `blush` | 音色、音频资产、温和提示 |
| `pink` | 失败、危险、警告；不作无意义装饰 |

柔和色块必须同时配合文字或图标，不以颜色作为唯一状态线索。

## 4. 字体与字号

- 全局使用 MiSans，缺失时回退 `Noto Sans SC` 与系统无衬线字体。
- 页面标题：28–32px，`font-medium`，紧凑行高。
- 内容对象标题：22–26px，允许换行，不能仅以省略号隐藏关键信息。
- 区块标题：14–17px，`font-medium`；色点只用于内容类型识别。
- 阅读正文：14–16px，行高 1.8–2.0。
- 工具正文与按钮：12–13px，行高至少 1.5。
- 标签与元数据：最小 11px。仅非关键装饰编号可使用 10px；不新增 9px 文本。
- 全大写与宽字距仅用于极短状态/分类，不用于中文长标签。

### 4.1 语义排版角色

业务组件优先使用以下语义 class，不再为同类文字重复拼装字号、颜色和行高：

| Class | 角色 | 默认表达 |
| --- | --- | --- |
| `ui-page-title` | 页面标题 | 28–32px、600 weight、紧凑行高 |
| `ui-section-title` | 区块标题 | 15–16px、600 weight，允许自然换行 |
| `ui-body` | 工具正文与说明 | 13px、1.72 行高、次级墨色 |
| `ui-reading-body` | 成稿、摘要、逐字稿等长文 | 15px、1.9 行高、48rem 阅读上限 |
| `ui-metadata` | 时间、数量、来源、状态补充 | 11px、1.55 行高，不承载主要结论 |
| `ui-control-label` | 按钮、筛选、表单 label | 12px；紧凑型最低 11px |

### 4.2 内容创作阅读顺序

- 一个阅读区先回答“这是什么”，再给出“最重要的结论”，最后提供证据、出处和动作。
- 标题、结论、正文、元信息至少形成三档清晰视觉重量；不得只靠卡片底色区分内容。
- 成稿、摘要和逐字稿保持开放段落与稳定行宽，不用 `line-clamp` 或固定高度裁切主要文字。
- 工具提示和状态可紧凑，用户需要理解、判断或校对的文字必须使用阅读正文层级。
- 用户粘贴材料固定标注“未自动核验”；自填 URL 固定标注“未抓取 / 未核验”，不能用“出处”视觉暗示系统已验证。AI 候选、AI 推断、本地创作者备注与来源摘录使用不同标题和状态，不能只靠颜色区分。
- exact Revision 选择必须同时给出正文预览与 provenance；AI 草案状态在标题、里程碑和输出区保持一致，未经显式人工另存不得显示成已确认成稿。

## 5. 宽度与页面框架

- 默认工具页面最大宽度：`72rem`（Tailwind `max-w-6xl`）。
- 音色资产与研究工作区：最多 `80–90rem`，内部仍需分区。
- 长文本阅读列：`48rem`（约 768px）以内。
- 页面水平 padding：移动端 16px，`sm` 起 24px。
- 页面纵向区块间距：16–24px；同一区块内部 8–16px。
- Sidebar 保持「工作台 / 内容库 / 音色库 / 自动化 / 设置」，桌面 256px、窄屏 72–80px；不得挤压主内容产生横向滚动。

## 6. 卡片层级

| 层级 | 用途 | 表达 |
| --- | --- | --- |
| 页面区块 | 独立任务、主要资产、主要阅读模块 | `bg-white/80`、18px 圆角、细边框、轻阴影、20–24px padding |
| 内部内容区 | 同一任务内的分组或工具区 | `bg-white/50–60`、12–16px 圆角、细边框、12–16px padding |
| 行项目 | 可扫描记录、紧凑资源 | 以分隔线/浅背景为主，默认不加阴影 |

不允许三层以上卡片嵌套；同一长页面不应让所有模块拥有相同视觉重量。阅读内容优先用开放式单列和分隔线。

## 7. 按钮

- 主操作：`lemon` 或 `sage` 实底，40px 左右高度。
- 编辑/转换：`lilac` 实底或轻底。
- 次操作：浅纸色/白色底 + 边框，或明确的文字按钮。
- 危险操作：`pink` 轻底或实底，必须有文字确认。
- 图标按钮：至少 36×36px，必须有 `title`/`aria-label`。
- 所有按钮有 `active:translate-y-px` 或等价即时按压反馈；不使用 hover 浮起。
- disabled 保持可读标签，并以透明度与 cursor 共同表达；loading 保留按钮宽度并显示阶段文字。

## 8. 输入框、选择器与筛选区

- 文本输入 40px 左右高度、12px 圆角；选择器可使用 pill，但最小字号 11px。
- label 必须常驻，不使用 placeholder 替代字段名。
- focus-visible 使用统一 2px `lilac`/深天蓝轮廓，不能只改变低对比度边框。
- 筛选区按“搜索 → 人物/类型 → 排序 → 结果数”排列；移动端允许换行，不设置不可收缩固定宽度。
- 已选状态同时使用背景、边框和语义属性（`aria-current`/`aria-pressed`）。

## 9. 状态标签与任务反馈

- Pill 最小 11px，包含文本；状态色映射：排队 `lemon`、处理中 `lilac`、完成 `sage`、失败 `pink`。
- Loading：页面与内容区使用骨架；按钮使用稳定文本或细进度反馈，不使用无限 spinner。
- Progress：显示阶段文字、百分比与确定型进度条；未知进度显示阶段，不伪造百分比。
- Error：说明发生了什么，并提供重试或恢复动作；抖动最多一次且 reduced-motion 下移除。
- Success：就地确认结果和下一步，不用大幅弹跳。普通保存使用短促勾选、边框或文案反馈；只有低频、可验证的创作里程碑可以使用局部庆祝动效。
- 长任务只要仍有 SSE、heartbeat 或持久进度就保持真实运行态，不用固定短墙钟制造超时；失败在 loading 清空后仍就地可见，未保存草稿不会因完成事件消失。
- Empty State：说明为什么为空、用户能做什么，并提供一个明确入口；避免只有“暂无数据”。

### 创作里程碑反馈

- 只反馈已真实持久化、可核验的关键阶段：第一份非空原文快照、第一条采用证据、第一版可审阅提纲、第一版带合法引用的初稿；不奖励模型调用次数、排队、HTTP 202 或单纯点击按钮。AI 草案落盘只表示已有可审阅资产，不得文案暗示用户已接受或内容已定稿。
- 反馈隐喻可使用编辑部盖章、证据链连接、结构搭建和小型纸屑 / 烟花，但保持在结果卡附近，8–12 个局部元素，不覆盖全页、不阻塞点击或焦点。
- 触发信号必须来自服务端事务提交后的唯一事件 ID。刷新、SSE 重连、重复 complete、幂等请求重放和前端列表数量变化不得重复播放。
- 反馈可关闭、可打断、无声音、无震动，不引入积分、等级、签到、连续天数或制造压力的 FOMO。
- `prefers-reduced-motion` 下只展示静态 `sage` 完成卡与清晰文案，不使用粒子位移、缩放或持续动画。

## 10. 长内容、摘要、逐字稿与观点

- 摘要保持单列阅读空间，正文最大 768px，14–16px 字号、1.8–2.0 行高。
- 逐字稿的嵌入式和全屏模式复用同一阅读器；搜索、说话人筛选、结果计数和键盘焦点保持一致。
- 元数据是上下文，不与摘要争夺首要视觉层级；默认以紧凑定义列表展示，编辑时展开表单。
- 观点卡先展示问题、核心观点、说话人、价值与证据时间；收藏、隐藏和详情始终可见但弱于观点正文。
- 1024–1280px 不使用可能裁切的绝对定位横向展开；观点使用可重排网格。卡片内容不得依赖 hover 展开。
- 观点详情打开证据后保留返回上下文；焦点返回触发元素。

## 11. 音频播放器与音色卡

- 所有播放器复用 `AudioPlaybackBar`；播放/暂停、seek、当前时间、总时长和可选倍速有稳定位置。
- 播放按钮至少 36×36px；range 控件必须可键盘操作，并有可见 focus。
- 波形仅表达可 seek 的进度，不持续做无信息价值的脉动；播放状态由按钮、进度和文字共同表达。
- 音色卡固定展示名称、类型、描述和试听入口；选择状态明确，试听不改变选择。
- 音色比较优先可扫描网格，不以过度截断描述换取整齐；移动端单列。
- 生成状态与其他长任务共用排队、处理中、完成、失败语义，并提供重试/下载/继续编辑入口。

## 12. Modal 与全屏工作区

- 全部使用 `ModalShell`。Dialog 以 opacity + 4–8px 位移/轻微缩放进入，180–220ms；transform-origin 对应触发上下文时优先设置。
- 全屏工作区只淡入，不使用明显缩放；Header 固定、内容独立滚动。
- 打开后焦点进入，Tab 被约束，Esc/关闭后焦点返回原触发点。
- 移动端 Dialog 接近全宽，保留 12–16px 安全边距；内容不能被 footer 遮挡。

## 13. 列表、表格与内容库

- 优先强化文件名/标题，其次是类型、状态、更新时间，再次是摘要与操作。
- 桌面可用多列；移动端转为堆叠行，不依赖横向表格滚动完成主要任务。
- 行操作在键盘聚焦和触屏下始终可见，不只在 hover 出现。
- 重复记录使用轻分隔和浅背景，避免每行都是高阴影白卡。

## 14. 动效

- Token：`fast 120ms`、`normal 180ms`、`slow 240ms`；常规 UI 不超过 300ms。
- Easing：进入 `cubic-bezier(0.22, 1, 0.36, 1)`；退出 `cubic-bezier(0.4, 0, 1, 1)`。
- 只过渡实际变化属性：`color`、`background-color`、`border-color`、`opacity`、`transform`、`width`；禁止 `transition-all`。
- 高频导航、Header、筛选和列表不做入场动画；Modal/Popover/显著状态切换可使用短动画。
- 不使用 `scale(0)`、持续装饰波形、每张卡片 hover 浮起或长列表交错入场。
- 动画必须可中途打断。`prefers-reduced-motion` 下移除 transform 位移、缩放和持续动画，但保留颜色、边框和短透明度反馈。
- 低频里程碑动效是上述高频稳定原则的受控例外；它必须由真实持久化成功触发，并遵循第 9 节的范围、去重和无障碍约束。

## 15. 响应式

- 移动端优先单列；控件可换行，主要按钮在需要时全宽。
- 1024px 不能假设有宽阔桌面空间：避免固定 280px+ 三列、绝对定位展开与大面积横向滚动。
- 1280px 适合双列工具布局，长阅读列仍保持上限。
- 1440px 只增加边缘留白或辅助栏，不拉宽正文。
- 必须验证长文件名、长中文段落、极多标签、空/错/加载状态均不造成横向意外滚动。

## 16. 可访问性

- 所有交互可用键盘完成；焦点可见且不被 sticky/fixed 区域遮挡。
- 图标按钮有可访问名称；状态变化使用适当的 `aria-live`。
- 点击目标至少 36×36px，主要移动端目标建议 44px。
- 正文与背景满足 WCAG AA；避免 `ink-soft/30–40` 承载必要信息。
- 不只用颜色、hover、动画或位置表达含义。

## 17. 明确禁止

- 纯黑/纯白主题、霓虹、大面积渐变、玻璃拟态和典型 AI 紫色渐变。
- 营销页式 hero、夸张大字、装饰性指标墙。
- `transition-all`、全页交错入场、每卡 hover 浮起、超过 300ms 的常规动画。
- 小于 11px 的关键文字、过度 `line-clamp`、固定高度裁切主要内容。
- 重复实现 Modal、音频播放器或业务数据结构；引入大型 UI 框架。
