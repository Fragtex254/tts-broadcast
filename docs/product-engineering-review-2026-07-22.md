# TTS Broadcast 产品与技术综合 Review

**Review 日期**：2026-07-22
**Review 范围**：hcds-studio 仓库全量（backend/ 51 个服务文件、12 个路由文件、551 行 schema；frontend/ 10 个页面、9 个 Zustand slice、54 个测试文件；AGENTS.md / DESIGN.md / docs/ 全套文档与 skill 体系）
**方法说明**：本 Review 基于完整代码走读、文档核对与数据流追踪完成，未在浏览器中实际点击运行。涉及"用户首次体验"的判断部分基于界面结构、空状态文案与路由组织推断，已标注。

## 一、独立理解：这个产品是什么

仅看代码与界面结构，不看 README，我的理解是：

1. **它是什么**：一个本地优先的内容创作工作台，主线是「内容项目（Brief → 来源 → 证据 → 提纲/主稿 Revision → 可选口播 TTS）」，同时内嵌了三套较强的研究型能力：音视频转录、播客结构化整理（Speaker/Segment/Turn/总结）、跨播客观点研究。
2. **服务谁**：一个人运营 AI 方向自媒体（公众号/小红书/播客）的创作者本人。所有设计（无鉴权、SQLite、单进程、API Key 存本地）都指向单用户本地工具。
3. **为什么用它而不是替代方案**：证据链（Evidence/Citation 由后端事实派生、AI 推断逐段标注、第一人称经验拒绝）和可核验性是真差异化，市面上没有现成工具做这件事；TTS 播报本身反而是可替代的。
4. **最核心场景**：把播客/素材/资讯变成一篇带可靠引用的成稿，顺带可以生成口播音频。
5. **当前最主要的价值主张**：**可核验的 AI 辅助创作**，而不是 README 里写的"AI 新闻播报"。
6. **无法快速理解的地方**：
   - README 第一屏讲的是"每日 AI 资讯 → 口播音频"，但顶级导航是「工作台 / 内容库 / 音色库 / 自动化 / 设置」，第一屏叙事和实际产品重心已经脱节。
   - "内容库"里混着播报历史、转录聚合、观点研究、内容项目四类不同抽象，新用户（或三个月后的你自己）很难一眼说明这四者的关系。
   - "自动化"是一个实际不可用的功能（见 P0-3），但它占据顶级导航的一个席位。

**结论**：产品真实重心（证据驱动创作）与对外表达（TTS 播报工具）已经不一致。这不是文案问题，是定位问题，下文第二节展开。

## 二、产品层面 Review

### 核心判断

**这个项目现在是两个产品共用一个外壳**：产品 A「AI 新闻口播工具」（broadcast/segments/音色/定时），产品 B「证据驱动内容研究创作台」（transcript/claims/evidence/artifact）。两者共享 LLM/TTS 基建，但用户心智完全不同：A 是消费型流水线，B 是研究型工作区。从代码量、schema 复杂度和文档着墨看，B 已经是实际主线（content_* 表族 + transcription_* 表族占了 schema 的 70% 以上），A 退化为 B 的"Render 出口"之一——project-facts 自己也承认 `broadcasts`/`segments` 是"兼容期的音频 Render，不得重新成为内容正文的唯一真相"。

### 逐条回答

1. **最核心的产品问题**：定位漂移没有在产品层收口。工作台首页、README、导航仍在为 A 服务，而真正的差异化在 B。结果是：新用户按 README 进来找"播报"，老用户实际在 B 里做研究，两套信息架构互相稀释。
2. **必要的功能**：内容项目主线（Brief/来源/证据/提纲/主稿）、转录（作为来源入口）、TTS Render（作为出口）、设置页。这些是闭环。
3. **价值较低甚至干扰的功能**：
   - **自动化（定时播报）**：当前没有执行器，保存即停用，启用返回 409——它是一个"看起来能用"的死功能，比没有更伤害信任。
   - **AI HOT 资讯抓取**：它是 A 产品的起点，但如果主线转向 B，它只是一个来源适配器，不该占据工作台的默认叙事位置。
4. **只能保留一个核心能力**：**证据链创作**（来源 → 证据 → 带引用成稿）。TTS、转录、观点研究都可以是它的输入/输出适配器。
5. **下一阶段最值得验证的假设**：「可核验引用 + AI 推断显式标注」是否真的是目标用户（自媒体创作者）愿意付出更高操作成本去换的价值，还是只是作者自己的品味。建议用 3-5 个真实用户走一遍"从播客到成稿"全流程验证。
6. **该加功能还是减功能**：**减**。当前功能密度已超出单用户产品的可理解边界，再叠加功能前应先做信息架构收敛。
7. **自以为重要但用户可能不在意的**：创作里程碑庆祝动效（纸屑/盖章）。规范写了整整一节（DESIGN.md §9），工程实现（唯一事件账本、防重放）成本不低，但它服务的是作者的产品审美，用户大概率无感。另：跨播客观点"关系分析"（claim_relations）是很重的功能，使用频率存疑。

### 当前最应该关注的产品指标

- 单用户维度：**每周完成"来源 → 显式保存成稿"完整闭环的次数**（不是 AI 调用次数、不是生成次数）。这是唯一验证价值主张的指标。当前产品内没有任何埋点能回答这个问题。

## 三、用户流程与交互 Review

### 核心流程：从素材到成稿（基于代码走读）

| 环节 | 现状 | 困惑点 | 放弃风险 |
|---|---|---|---|
| 1. 首次启动 | 任何功能前必须先去设置页填 LLM Key + TTS Key | 启动后首页没有"先去配置 Key"的引导式卡位（需实测确认）；Key 未配置时各入口的错误提示依赖运行时 400 | **高**：第一步就要求两个外部账号 Key |
| 2. 导入素材 | 转录页上传/批量转录 → 转录聚合 → 进入内容项目关联来源 | "转录结果"和"内容项目来源"是两个概念，需要用户理解"导入/关联"动作 | 中 |
| 3. 证据提取 | AI 提取 → 用户逐条 candidate/selected/rejected | 决策态与生命周期分列是对的，但用户需要理解两套状态词 | 中 |
| 4. 提纲/主稿 | AI 草案 → 显式人工保存才成为 Revision | 这是正确设计，但"AI 草案 ≠ 成稿"的心智需要界面反复强调 | 低 |
| 5. 口播 Render | Revision → Broadcast → 分段 TTS | 整篇/分段两个模式 + 编辑器页是上下文页（刷新丢状态，见下） | 中 |

### 特别指出的五项

1. **用户最容易流失的三个位置**：① 首次启动填 Key；② "转录结果 → 内容项目来源"的概念跨越；③ 分段编辑器——它依赖全局 store 传上下文，**直接刷新 `/editor` URL 会丢状态**（前端探查确认），长任务中途刷新后用户找不到回到任务现场的路。
2. **认知成本最高的环节**：转录聚合里的四层概念（Transcript/Speaker/Segment/Turn）+ 总结 + 观点。CONTEXT.md 定义了精确的术语边界，但这些术语是普通用户不会自然理解的工程语言。
3. **最应该简化的流程**：批量转录 → ZIP 导出 → 再导入项目这条路径，和"单文件转录 → 直接成为来源"应该统一成一个"素材入库"心智。
4. **最缺少反馈的操作**：SSE 任务中用户离开页面后，连接只能等 complete/error 自然关闭（`transcribeSlice.ts:236,420,497`，client 实例是局部变量），**回到页面时没有"任务仍在后台运行"的全局指示**。
5. **最不符合直觉的交互**：`/editor` 和转录上下文页靠 Zustand 全局态传参进入，违反"URL 即可恢复现场"的直觉；自动化页"保存成功但实际不可用"（虽返回 409，但一个永远在导航里的不可用功能本身就是反直觉的）。

### 状态完整性（做得好的部分）

长任务的 loading/error/空态、SSE 开始/进度/完成/失败、lease 过期收敛为可重试失败态、失败原因落到具体段落（`segments.error_message`）——这套长任务反馈体系是同体量项目里少见的完整，应保留。

## 四、UI 与视觉设计 Review

DESIGN.md 是一套成熟度很高的设计系统（语义色、排版角色、卡片层级、动效 token、reduced-motion 约束），且**执行到位**：前端探查确认 tsx 中 0 个硬编码 hex、0 个 Tailwind 默认色板，`fixed inset-0` 仅存在于 ModalShell，`<audio>` 仅存在于 AudioPlaybackBar。这种执行纪律在单人项目里罕见，是资产不是负担。

### 五个最影响专业感的视觉问题

1. **Sidebar 顶级导航里有一个死功能**（自动化），视觉权重与可用功能相同——比任何视觉瑕疵都更伤专业感。
2. **内容库混合四类对象**（播报/转录/观点/项目），列表的视觉层级（DESIGN.md §13 要求"标题 > 类型 > 状态 > 时间"）在不同对象间语义不一致，"类型"一列承担了过重的区分任务。
3. **批量状态仅靠色点**（BATCH_STATUS_DOTS），违反 DESIGN.md §3 自己的"不以颜色作为唯一状态线索"和 §16 可访问性条款——设计系统在自己家里破了例。
4. **工作台首页叙事**：默认首屏服务于"AI 资讯 → 播报"的旧主线，与实际产品重心不符（同定位问题在视觉层的投射）。
5. **ErrorBoundary fallback 只有"刷新页面"**（`ErrorBoundary.tsx:51`），无错误说明、无 `role="alert"`，全站崩溃时的视觉呈现与整体设计质感断层。

### 审美问题 vs 可用性问题

- 审美：暖纸色系、Soft Editorial 气质与"研究型工作台"匹配，无模板感、无 AI 紫色渐变，判断正确。
- 可用性：上述 2/3/5 都是可用性问题而非审美问题。
- **最应保留的视觉设计**：语义色 token 体系与 ModalShell/AudioPlaybackBar 的强制收敛。
- **最应删除的装饰**：里程碑庆祝动效的优先级应降到所有可用性问题之后（它不是错误，是投入时序问题）。

## 五、前端工程 Review

### 逐条回答

1. **最可能首先失控的地方**：`Transcribe.tsx`（942 行，约 25 个 useState + 20 个 store 选择器）和 `transcribeSlice.ts`（872 行）。转录是当前迭代最活跃的功能，而它的宿主已经是全站最大的上帝组件+上帝 slice，继续堆功能会先在这里崩。
2. **能运行但不适合扩展的实现**：
   - 服务端状态与 UI 状态混在单一 Zustand store（925 行 `AppState` 大接口），无失效/重取策略，全靠手动 fetch。功能再长一倍，"哪份数据是新鲜的"会变成日常 bug 源。
   - SSE client 实例是 slice 内局部变量，无 store 级注册表，无法统一断开/重连/全局展示。
3. **明显技术债**：
   - SSE 事件 payload 全部 `JSON.parse(...) as Type`（`sseClient.ts:92,98,105`）——全站契约最薄弱的缺口，后端改字段时前端零防护。
   - `segmentSlice.ts:65-137`、`presetSlice.ts:18,30` 直接信 `response.data`，绕过了项目自己建立的 Zod 校验纪律。
   - 跨 slice 直接读取：`segmentSlice.ts:41,112,135` 读 `get().voiceConfig`。
4. **不必要的抽象**：基本没有。依赖极精简，没有为抽象而抽象的层。这是好评。
5. **需要尽快补测试的地方**：`sseClient.ts`（无测试 + 无校验，双重裸奔）、`api.ts`（仅 1 个测试）、Transcribe 与 Settings 两个最大页面无组件级测试。
6. **直接影响用户体验的问题**：editor/transcribe 上下文页刷新丢状态；SSE 无自动重连；ErrorBoundary 无法局部恢复。
7. **功能增加三倍最先崩的模块**：转录域（组件+slice）→ 其次是 store 的大接口（所有域共享一个 `AppState`，merge 冲突和认知负担会指数上升）→ 然后是观点卡列表（`TranscriptClaimsPanel.tsx` 无虚拟化，claim 数百条即卡顿）。

### 亮点

路由级 `React.lazy` 全覆盖、自研虚拟化长列表（useVirtualTranscriptTurns）、108 处防重复提交、ModalShell 焦点管理完整、非测试代码 0 裸 `any`。前端工程质量整体高于同体量个人项目的中位数。

## 六、后端与系统设计 Review

### 逐条回答

1. **当前最大的后端风险**：**API Key 明文存 SQLite，且 `GET /api/settings` 原样明文回传全部密钥**（`routes/settings.js:13-25`）。配合全站无鉴权 + 开发 CORS 全放开（`app.js:13-23`），意味着：同一局域网任何人、浏览器里任何一个恶意网页（配合 DNS rebinding 或直接请求 localhost:3001）、或任何能读到 `backend/data/` 的进程，都能拿走全部 Key。个人本地工具可以接受"无鉴权"的取舍，**不能接受"明文回传"**——脱敏返回（`sk-***` + 单独测试端点）是半天工作量的必修项。
2. **安全隐患清单**：见 P0-1/P1；另 `transcribe.js:89` 把内部 `error.message` 直接回给客户端；`voicePresets.js` 上传无 fileFilter。
3. **用户量上升会出问题的地方**：本项目定位单用户，此问题换成"数据量上升"：① 观点搜索全表扫 embedding 在 JS 里算余弦（`researchService.js:25-34`）+ N+1（`:47,:50-55`），claim 上千条后搜索明显变慢；② `broadcasts.saved` 无索引但高频过滤（`broadcastStore.js` 6 处）；③ 列表接口大文本字段 SELECT *。当前数据量下都不阻塞，属于"知道在哪、暂不修"的可接受债。
4. **应该重构 vs 不用动**：应修——settings 密钥脱敏、进程级异常兜底；**不用动**——rateLimitedQueue（971 行，复杂但有测试和持久化账本支撑，是核心资产）、mimo.js 1278 行（丑但稳定，拆它是中成本中收益，排到功能空窗期）。
5. **可能导致数据错误/线上事故的**：无 `uncaughtException`/`unhandledRejection` 兜底，异步 runner（`transcriptionClaimRunner.js:12` setInterval）内一个未捕获异常即可杀进程；内存队列崩溃丢任务有 lease 表兜底（设计正确），但进程死了 lease 恢复要靠下次读取触发，期间无监控可知。
6. **过度设计的部分**：**没有明显的过度设计**——这是难得的评价。generation_token CAS、幂等账本、lease、里程碑唯一事件，每一个都对应一个真实发生过的并发/重放问题（从文档措辞能看出是被 bug 教育出来的），复杂度有其债务凭证。
7. **架构是否适合现阶段**：适合。单进程 + SQLite + 内存队列 + WAL + lease 恢复，对单用户本地工具是正确档位。

### 测试与契约

后端 51 个测试文件约 563 用例，队列、CAS、幂等、lease 等关键路径全部有覆盖——这是全项目最硬的一块。缺口：`routes/sse.js`、`contentWorkspace.js` 无直接路由测试。

## 七、前后端协作与整体系统 Review

1. **边界最混乱的地方**：转录域。转录既是独立功能（转录页、历史、ZIP 导出）又是内容项目的来源入口，同一批 `transcription_results` 数据被两套心智消费，前后端都因此付出了概念税（前端要维护"导入编辑器"的跨 slice 写入，后端要在删除时查项目引用）。
2. **最影响开发效率的协作问题**：SSE payload 无共享类型/schema。后端 SSE 事件是字符串协议，前端用 `as` 断言接收，这是端到端契约链上唯一的裸奔段——而 SSE 恰恰是本项目长任务的生命线。
3. **最需要统一的规范**：分页/列表协议。三套并存：`{broadcasts, pagination}`（page/limit/offset）vs `{results}`（仅 limit）vs research 的 limit 截断。
4. **最容易产生 Bug 的数据流**：「刷新/多标签/重试 → 幂等键 → lease → CAS 收口」这条链。它现在有完备防护和测试，但每加一个新的长任务类型都要重新正确实现一遍全链，缺一个"长任务脚手架"把 lease+heartbeat+token+SSE 事件固化成模板，否则下一个长任务（比如未来的关系分析批量任务）就是下一个 bug 温床。
5. **最值得优先重构的端到端流程**：**设置与密钥管理链路**（存储 → GET 脱敏 → 测试连接 → 错误状态归类）。它不是最复杂的，但风险/成本比最高。

## 八、工程化与团队协作 Review

这个项目的工程化水平是其最强的维度之一：AGENTS.md + skill 路由表 + project-facts + BACKEND/FRONTEND_CONVENTIONS + DESIGN.md 构成的文档体系、GitHub Actions PR 门禁（后端测试 + 前端 lint/build）、563 后端用例 + 54 前端测试文件、不可协商铁律显式成文。

1. **新工程师接手最大的困难**：不是缺文档，是**文档太多且术语密度极高**——AGENTS.md 的 Hard Rules 有 20+ 条，新人第一周无法分辨哪些是"会被 CI 拦的"、哪些是"约定俗成的"。缺一份一页纸的"第一周生存指南"。
2. **最缺失的工程基础设施**：① `/health` 健康检查端点；② 优雅停机（`scheduler.shutdown()` 存在但无信号注册）；③ 错误监控（pino 日志写文件，但没有任何聚合/告警路径，进程半夜死了无人知晓）；④ 前端无 CI 测试步骤（CI 只跑 lint+build，54 个测试文件不在门禁里）。
3. **值得建立的规范**：SSE 事件协议 schema 化（前后端共享）；长任务实现 checklist。**没必要的规范**：不建议再为单人项目增加更多流程性文档。
4. **依赖人工易出错的流程**：新增长任务类型时手工复刻 lease/token/SSE 全链；发布无版本号/无回滚（单人项目可接受，但 start.sh 启动即代表"发布"）。
5. **是否过度工程化**：接近临界但尚未越线。skill 体系本身是克制的（按需加载），风险在于文档维护成本随功能线性增长，单人长期维持这套文档纪律本身就是负担——建议把 AGENTS.md Hard Rules 控制在 15 条以内，超出部分下沉到 skill。

## 九、问题优先级清单

### P0 — 阻塞问题

**P0-1 密钥明文存储并经 API 明文回传**
- 位置：`backend/src/routes/settings.js:13-25`、`settings` 表
- 描述：所有 LLM/TTS/Embedding API Key 明文存 SQLite，`GET /api/settings` 原样全量返回
- 证据：后端探查确认无脱敏逻辑；前端设置页直接渲染返回值
- 影响：密钥泄露 = 资金损失；任何能访问 localhost:3001 或数据文件的一方即得全部 Key
- 严重程度：Critical
- 修改方向：GET 返回脱敏（`sk-***abcd` + `is_set` 标记）；写入走单独端点且不回显；可选加密存储（机器级 keychain 或对称加密 + 本地 passphrase）
- 成本：低（脱敏）/ 中（加密存储）
- 立即处理：**是** | 需验证：否

**P0-2 全站无鉴权 + 开发态 CORS 全放开**
- 位置：`backend/src/app.js:13-23`
- 描述：无任何认证中间件，`origin: true` 反射所有来源
- 影响：局域网/恶意网页可调用全部 API（含上传、删除、LLM 消费）——与 P0-1 叠加形成完整攻击链
- 严重程度：Critical（若仅限 127.0.0.1 绑定可降级为 P1）
- 修改方向：默认绑定 127.0.0.1；CORS 白名单化；如需局域网访问则加一层简单 token（启动时生成，前端注入 header）
- 成本：低 | 立即处理：**是** | 需验证：确认实际部署形态

**P0-3 "自动化"是不可用功能却占顶级导航**
- 位置：`backend/src/services/scheduler.js`、前端 `/automation` 路由与 Sidebar
- 描述：无执行器注入，保存即停用、启用返回 409，README 却仍把"定时播报"列为核心能力
- 影响：产品信任损伤 + 导航噪音 + 文档与实现矛盾（违反项目自己的"文档优先"红线）
- 严重程度：High（产品层面阻塞"信息架构收敛"）
- 修改方向：二选一——① 导航隐藏 + README 移除，代码保留为 backlog；② 注入真实执行器补齐功能。当前状态下**推荐 ①**
- 成本：低 | 立即处理：**是** | 需验证：否

### P1 — 高优先级

**P1-1 无进程级异常兜底与优雅停机**
- 位置：`backend/src/app.js`（无 uncaughtException/unhandledRejection/SIGTERM 注册）；`transcriptionClaimRunner.js:12` setInterval 内异常可杀进程；`scheduler.shutdown()` 存在但无人调用
- 影响：异步任务一个异常 = 进程死亡 + 进行中队列任务丢失 + 无任何告警
- 修改方向：注册兜底（记日志 + 退出码）；SIGTERM/SIGINT → 停 scheduler → 关 SSE → 关 DB
- 成本：低 | 立即处理：是

**P1-2 定位漂移未收口：README/首屏叙事 ≠ 实际产品重心**
- 位置：`README.md`、工作台首页、导航信息架构
- 描述：见第二节。两个产品共壳，表达层仍是旧的
- 修改方向：明确主线为"证据驱动内容创作"，README/首屏重写，TTS 播报降级为出口能力
- 成本：中（主要是决策成本） | 立即处理：是（先决策后动手）

**P1-3 Transcribe 域上帝组件 + 上帝 slice**
- 位置：`frontend/src/pages/Transcribe.tsx`（942 行）、`store/transcribeSlice.ts`（872 行）
- 修改方向：按交互单元拆（上传区/批量面板/历史面板/预览），slice 按"单任务/批量/结果"拆三个
- 成本：中 | 立即处理：否（下个功能迭代前先拆）

**P1-4 SSE 契约裸奔 + 生命周期无全局管理**
- 位置：`frontend/src/services/sseClient.ts:92-105`（`as` 断言、无测试）；client 实例为 slice 局部变量，页面卸载不清理、无重连、无"后台任务进行中"全局指示
- 修改方向：SSE 事件前后端共享 Zod schema；client 注册表入 store；断线指数退避重连；顶部全局任务条
- 成本：中 | 立即处理：schema 先行（低成本）

**P1-5 `/editor` 与转录上下文页刷新丢状态**
- 位置：React Router 结构 + 跨 slice 传参（`Transcribe.tsx:69-70`）
- 影响：长任务中途刷新即迷路；违反 URL 可恢复直觉
- 修改方向：上下文进 URL（`/editor/:broadcastId`、`/history/transcriptions/:id` 已是正确模式，editor 对齐之）
- 成本：中 | 立即处理：否（随 P1-3 一起做）

**P1-6 内部错误消息直接回传客户端**
- 位置：`routes/transcribe.js:89`（`error.message` 直出）；`settings.js:100` 同类错误一处 200 一处 400 不一致
- 修改方向：统一错误映射表（外部 API 错误已做，内部错误补齐）；测试端点错误码归一
- 成本：低 | 立即处理：是

## 十、最终结论

### 1. 一句话总评

**这是一个工程素养明显高于定位清晰度的项目：并发一致性与证据链设计是准专业级的，但它还没决定自己到底是谁——而密钥安全是它现阶段唯一可能真正伤到自己的地方。**

### 2. 当前最严重的三个问题

1. **密钥明文存储 + 明文回传 + 无鉴权 + CORS 全放开**（P0-1/P0-2 叠加）：唯一可能造成真实金钱损失的问题，半天能修掉大头。
2. **产品定位漂移未收口**：证据创作是实际主线，播报是旧叙事，README/首屏/导航/死功能导航位都在为旧叙事交租。
3. **转录域前端复杂度失控前夜**：942 行页面 + 872 行 slice + 最活跃迭代域，三者叠加，下一个功能就会付出代价。

### 3. 最值得优先修改的三个地方（按投入产出比）

1. 设置页密钥脱敏 + CORS/绑定收敛（一天，消除 P0）
2. 自动化导航隐藏 + README 重写对齐真实主线（一天，消除最大的表达矛盾）
3. SSE 事件 schema 化 + client 注册表（两天，补上契约链最薄弱环 + 顺手解决重连/全局任务条）

### 4. 最值得保留的三个地方

1. **证据链架构**：Segment 不可变事实 / Turn 派生校对 / Evidence 后端派生摘录 / decision_state 与 lifecycle_status 分列 / AI 推断显式标注——这是产品真正的护城河，任何重构不得稀释。
2. **长任务一致性体系**：lease + heartbeat + generation_token CAS + 幂等账本 + 补偿清理，每一层都有对应测试。这套模式是从真实 bug 里长出来的，继续作为所有新长任务的模板。
3. **设计系统执行纪律**：ModalShell/AudioPlaybackBar 单点收敛、0 硬编码颜色、语义 token 全覆盖。保持，别让未来的急活破窗。

### 5. 当前不建议做的事

- **不要做多用户/账号体系**：无鉴权是单用户本地工具的合理取舍，先把脱敏和绑定做掉即可。
- **不要引入 Postgres/消息队列/微服务**：SQLite + WAL + 内存队列 + lease 的档位正确，数据量证据不支持升级。
- **不要继续做里程碑庆祝动效的花样**：现有规范已够用，投入转到可用性问题。
- **不要急着拆 `mimo.js` 和 `rateLimitedQueue.js`**：它们丑但有测试有事实依据，拆它们是 P3。
- **不要给"自动化"补执行器**：除非定位决策结论是 A（播报）仍是主线——按我的判断不是。

### 8. 最终判断

- **当前阶段**：**超出 MVP 的"重度自用产品"**——不是 Demo（有真实持久化、幂等、恢复设计），也不是可发布产品（安全、定位、首启体验不达标）。对作者本人是"可用且每天都在用的工具"，对第二个用户是"装不上、看不懂、不敢填 Key"。
- **距离正式发布最缺什么**：按顺序——① 密钥安全底线；② 一个对外的、诚实的"这是什么"的答案；③ 首次启动 30 秒到第一个价值的引导路径。功能一个都不缺。
- **如果我是负责人，下一步只推一件事**：**把"证据驱动创作"立为唯一主线并改写所有对外表达**——因为定位一旦收口，自动化去留、README、首屏、内容库 IA、A/B 产品关系全部迎刃而解；而安全问题作为一天能修的 P0，不需要"推动"，直接修。
