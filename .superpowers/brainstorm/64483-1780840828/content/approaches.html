导航重构方案对比

将当前 3 个标签拆分为 4 个标签的三种设计方案

A

方案 A：最小改动拆分（推荐）

将现有 Dashboard 组件直接拆到两个新页面，保持后端不变

侧边栏 4 个标签：
📡 信源收集 → QuickGenerate 独立页面
✏️ 口播稿编辑 → ScriptPreview + VoiceGenerator + SegmentEditor + AudioPlayer
📋 历史记录 → 保持不变 + 新增"重新编辑"按钮
⚙️ 设置 → 新增 TTS API Key 输入框

数据流：改写完成 → 自动跳转 /editor，通过 Zustand store 传递 script 和 currentBroadcast

后端改动：仅需在 Settings 默认值中加入 mimo_tts_api_key

✅ 优势：改动最小、风险低、复用所有现有组件
⚠️ 劣势：页面间跳转增加一步操作
B

方案 B：流程步骤式

侧边栏加入步骤指示器，清晰展示工作流进度

侧边栏带进度指示：
❶ 信源收集（完成后显示 ✓）
❷ 口播稿编辑（完成后显示 ✓）
── 分隔线 ──
📋 历史记录
⚙️ 设置

额外设计：侧边栏显示当前步骤进度，完成后自动高亮下一步

✅ 优势：工作流一目了然，新手友好
⚠️ 劣势：增加状态管理复杂度，步骤指示器可能过度设计
C

方案 C：Tab 子导航

保留"工作台"为一级导航，内部用 Tab 切换信源/编辑

侧边栏 3 个标签：
🎙️ 工作台 → 页面顶部 Tab：[信源收集] [口播稿编辑]
📋 历史记录
⚙️ 设置

数据流：同一页面内 Tab 切换，状态天然共享

✅ 优势：侧边栏简洁，Tab 切换无页面跳转
⚠️ 劣势：没有真正拆分页面，"控制台聚合太多逻辑"的问题仍在
